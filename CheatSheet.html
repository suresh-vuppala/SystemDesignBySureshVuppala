⚡ 1 · FOUNDATIONS

System Design Framework

The structured approach to tackle any design interview (45-60 min)

1. REQUIREMENTS (5 min)  — Clarify scope. Ask questions. Define FR + NFR.
2. ESTIMATION (5 min)    — Users, QPS, storage, bandwidth (back-of-envelope).
3. HIGH-LEVEL DESIGN (10 min) — Draw: clients → LB → services → DB → cache.
4. DETAILED DESIGN (20 min)   — Deep dive 2-3 critical components.
5. TRADE-OFFS (5 min)    — Alternatives, bottlenecks, failure modes.
6. SCALING (5 min)       — How to handle 10x, 100x growth.

Functional vs Non-Functional Requirements

What the system does vs how well it does it

Functional (FR)Non-Functional (NFR)
User can post tweet, follow, view timelineLatency <200ms p99
User can upload/download filesAvailability 99.99% (52 min downtime/year)
User can search productsThroughput 10K RPS, Durability 11 nines
User can make paymentsConsistency (strong for payments, eventual for likes)
Core Challenges: Too many users → horizontal scaling, LB, caching. Too much data → sharding, tiered storage. Low latency → caching, CDN, geo-distribution. High availability → replication, multi-region, graceful degradation.

Scaling Basics

Vertical vs Horizontal — the fundamental scaling decision

Vertical (Scale Up)Horizontal (Scale Out)
Bigger machine (more CPU/RAM/IOPS)More machines behind a load balancer
Simple — no code changesUnlimited — add nodes as needed
Has ceiling — biggest machine has limitsComplex — distributed state, consistency
Single point of failureFault tolerant (node dies → others serve)
Guarantee: Horizontal scaling provides linear throughput growth — doubling nodes roughly doubles capacity, because each node handles an independent subset of traffic.

Stateless vs Stateful

Stateless services scale horizontally easily; stateful services need sharding/replication

StatelessStateful
No session data on server. Every request self-contained.Server remembers client state between requests.
Easy to scale — any instance can handle any requestHard to scale — need sticky sessions or shared state
Store state externally (Redis, DB, JWT)WebSocket connections, in-memory caches, game servers
Guarantee: Stateless services are horizontally scalable by design because no request depends on which server handles it — any server can serve any request identically.
Real-world: REST APIs are stateless. WebSocket servers are stateful (connection pinned to server). Best practice: externalize state to Redis/DB → make services stateless.

Serialization & Deserialization

Converting objects to bytes (serialize) and back (deserialize) for storage/transmission

FormatTypeSizeSpeedUse Case
JSONTextLargeSlowREST APIs, config files, human-readable
Protocol BuffersBinarySmallFastgRPC, internal services (Google)
AvroBinarySmallFastKafka events, Hadoop (schema in header)
MessagePackBinarySmallFastRedis, embedded systems
XMLTextVery largeSlowSOAP, legacy enterprise
Guarantee: Binary formats (Protobuf, Avro) provide schema evolution with backward/forward compatibility — old consumers can read new messages and vice versa, because fields are identified by number not name. This is critical for zero-downtime deployments.
Schema Registry (Confluent) — central store for Avro/Protobuf schemas. Enforces compatibility rules. Used with Kafka to prevent breaking changes.

Why it matters: L4 LB routes by IP+port (fast, no content inspection). L7 LB routes by URL/headers/cookies (smart, can do path-based routing). Most system design discussions happen at L4 and L7.

TCP / UDP

Transport layer — TCP (reliable, ordered) vs UDP (fast, unreliable)

TCPUDP
Connection-oriented: 3-way handshake (SYN→SYN-ACK→ACK)Connectionless: Send immediately
Ordered + guaranteed: Retransmits lost packetsBest-effort: No retransmission, may arrive out-of-order
Flow + congestion control: Sliding window, slow startNo flow control — sender controls pace
Guarantees: TCP guarantees ordered, reliable, duplicate-free delivery via sequence numbers, ACKs, and retransmission. UDP guarantees nothing — but that's why it's fast (no overhead).
TCP: Banking, HTTPS, DB connections, email. Ports: 80/443/3306/5432/6379/9092
UDP: Live video, gaming, DNS, VoIP, QUIC (HTTP/3). Ports: 53/123/443(QUIC)
Real-world: Fortnite/Valorant use UDP for player positions (few lost packets OK, low latency critical). QUIC (HTTP/3) — UDP-based with built-in TLS 1.3, 0-RTT resumption. Used by Chrome, YouTube, Cloudflare.

DNS

Translates domains → IP addresses. Resolution: Browser → OS → Resolver → Root → TLD → Authoritative

RecordPurposeExample
A/AAAADomain → IPv4/IPv6example.com → 93.184.216.34
CNAMEAliaswww → example.com
MXMail routing→ mail.example.com
SRVService discovery_http._tcp.example.com
Guarantees: DNS provides eventual consistency — changes propagate within TTL. Low TTL (60s) = fast failover but high query load. High TTL (86400s) = less load but slow propagation.
Real-world: Route 53 GeoDNS routes to nearest region. Cloudflare 1.1.1.1 resolves in ~11ms. Consul uses DNS for service discovery. Anycast — same IP from multiple locations.

🔌 4 · APIs & COMMUNICATION

REST API

Stateless HTTP-based — universal, cacheable. The default choice for public APIs

GET    /api/v1/products/123     → Fetch (cacheable, idempotent)
POST   /api/v1/orders           → Create (use idempotency key to prevent dupes)
PUT    /api/v1/orders/456       → Replace (idempotent)
DELETE /api/v1/orders/456       → Cancel (idempotent)

Pagination: ?cursor=abc123 (preferred) or ?page=2&limit=10
Caching: Cache-Control: max-age=3600 · ETag: "abc123"
Rate Limit: X-RateLimit-Limit: 1000 · X-RateLimit-Remaining: 847
Guarantees: Statelessness — no server-side session, any instance handles any request. Idempotency of GET/PUT/DELETE — safe to retry on failure. Cacheability — HTTP caching (CDN, browser) reduces load.
Real-world: Stripe API — gold standard (idempotency keys, versioning, pagination). GitHub API v3 — REST. Twilio — REST for SMS/voice.

gRPC

HTTP/2 + Protobuf — 10x faster than REST. 4 call types: unary, server-stream, client-stream, bidirectional

Guarantees: Type safety — .proto schema + codegen catches incompatibility at compile time. Deadline propagation — timeout flows through entire call chain. Multiplexing — multiple concurrent calls on single HTTP/2 connection.
Real-world: Google internal comms. Netflix/Uber microservice-to-microservice. Best for: internal APIs, 10K+ RPS, bidirectional streaming. Not for browsers (use gRPC-Web proxy).

GraphQL

Client specifies exactly which fields — single endpoint, no versioning, strongly typed schema

Guarantees: No over-fetching — client gets only requested fields. Schema contract — server validates queries against schema before execution. Introspection — clients can discover available types/fields.
Risks: N+1 problem (fix with DataLoader batching). Deep query DoS (fix with depth limiting + cost analysis). Caching hard (each query unique). GitHub API v4, Shopify Storefront use GraphQL.

API Extras

SOAP, idempotency, sync/async, CORS, documentation

TopicDetail
SOAPXML-based, WSDL contract, WS-Security. Enterprise/legacy (banking). Heavier than REST.
Idempotent APIsSame request N times = same result. POST isn't idempotent — use idempotency keys (Idempotency-Key: uuid) to prevent duplicate orders. Stripe pattern.
Sync → AsyncPOST → 202 Accepted + job ID → poll GET /jobs/{id} or receive webhook callback. Decouple with queue.
CORSBrowser security. Server sends Access-Control-Allow-Origin. Preflight OPTIONS for non-simple requests.
OpenAPI/SwaggerStandard REST spec. Auto-generates docs, SDKs, mock servers. Postman for testing.

Real-time Communication

Technologies for pushing data from server to client — choose based on direction + latency needs

TechDirectionLatencyBest ForGuarantee
Short PollingClient → Server (repeated)N secDashboard refreshSimple but 99% requests empty (wasteful)
Long PollingServer holds connection~secChat (pre-WS era)Near real-time but 1 conn/client held open
WebSocketFull duplex~msLive feeds, gaming, collab editingPersistent bidirectional — server pushes instantly. ~100K conn/server.
SSEServer → Client~msNotifications, feedsAuto-reconnect built into browser. Event ID for resuming. Text-only.
WebRTCPeer-to-peerUltra-lowVideo/audio callsDirect P2P — no server bandwidth for media. Browser-enforced encryption.
WebSocket Lifecycle:
  HTTP Upgrade → 101 Switching → Full Duplex (binary frames)
  Heartbeat: Ping every 30s to detect dead connections
  Reconnect: Exponential backoff (1s → 2s → 4s → 30s max)
  Scale: Use Redis Pub/Sub to fan out across multiple WS servers

Choosing:
  One-way server push, simple → SSE
  Bidirectional, low latency → WebSocket
  Video/audio P2P → WebRTC
  Simple, infrequent updates → Long Polling
Real-world: Slack — WebSocket for messaging. Figma — WebSocket for collab editing. Zoom — WebRTC for video. Robinhood — WebSocket for live stock prices. Socket.IO — auto-fallback to polling.
🏗️ 5 · INFRASTRUCTURE

Load Balancer

L4 (TCP — fast, IP+port) vs L7 (HTTP — smart, URL/headers). ALB, NLB, NGINX, HAProxy

AlgorithmHowBest For
Round RobinRotate sequentiallyEqual-capacity servers
Least ConnectionsRoute to fewest activeVarying request durations
IP HashHash client IP → consistent serverSession affinity
Consistent HashingHash ring, minimal disruptionCache servers
Guarantees: No single point of failure (if LB is HA). Health checks auto-remove unhealthy backends. SSL termination at L7 offloads crypto from backends.
Sticky Sessions: Route same client to same backend. Problem: hot spots, breaks on death. Better: externalize session to Redis.

API Gateway

Single entry point — auth, rate limiting, routing, SSL. Kong, Apigee, AWS API Gateway

ResponsibilityHow
AuthVerify JWT/API key before routing upstream
Rate LimitingPer-client/IP/tier quotas (Redis-backed)
RoutingPath-based forwarding to microservices
BFF PatternSeparate gateways for mobile vs web vs partner
Guarantee: API Gateway provides a single enforcement point for cross-cutting concerns — auth, rate limiting, logging happen once at the edge, not duplicated in every service.

Forward & Reverse Proxy

Forward = hide client (corporate proxy). Reverse = hide servers (NGINX, Cloudflare)

Reverse Proxy Guarantees: Backend isolation — clients never see internal IPs. SSL termination — decrypt at proxy, HTTP internally. Caching — serve cached responses without hitting origin. DDoS absorption at edge.

NGINX

Reverse proxy, LB, web server, API gateway. ~34% of all websites. Event-driven, non-blocking.

upstream backend {
    least_conn;
    server app1:8080 weight=3;
    server app2:8080;
}
server {
    listen 443 ssl;
    location /api/ { proxy_pass http://backend; }
    location / { root /var/www/html; }  # Static files directly
}
Why NGINX is fast: Event-driven architecture — single master + worker processes, each handling thousands of concurrent connections (vs Apache's thread-per-connection). Solves C10K problem.

Docker & Kubernetes

Docker packages apps into containers. Kubernetes (K8s) orchestrates them at scale.

Docker ConceptDetail
ImageImmutable template with app + dependencies. Built from Dockerfile. Stored in registry (Docker Hub, ECR).
ContainerRunning instance of image. Lightweight isolation (shared kernel, not full VM). Starts in seconds.
VolumePersistent storage that survives container restarts.
K8s ConceptDetail
PodSmallest unit. 1+ containers sharing network/storage. Ephemeral.
ServiceStable network endpoint for pods (ClusterIP, NodePort, LoadBalancer).
DeploymentDeclarative desired state. Rolling updates, rollbacks.
HPAHorizontal Pod Autoscaler — scale on CPU/memory/custom metrics.
StatefulSetOrdered, stable pod identities. For DBs, Kafka, ZooKeeper.
IngressHTTP routing rules (NGINX Ingress, Traefik). External traffic → services.
Deployment Strategies: Rolling (gradual, default) · Blue-Green (swap envs) · Canary (5% traffic first) · A/B (feature flags, Istio traffic split)
Guarantees: K8s guarantees desired state reconciliation — if a pod dies, controller restarts it. Self-healing via liveness/readiness probes. Service discovery via DNS (pod-name.service.namespace.svc.cluster.local).
Real-world: Google (Borg predecessor). Spotify 2000+ services on K8s. Managed: EKS (AWS), GKE (Google), AKS (Azure).

Service Mesh

Istio, Linkerd — sidecar proxy (Envoy) for service-to-service networking

Guarantees: mTLS everywhere (zero-trust). Automatic observability (metrics, traces per call). Traffic management (canary, retries, circuit breaking) via YAML, not code. vs API Gateway: Gateway = north-south (external→internal). Mesh = east-west (internal→internal).

Multi-Region & Multi-Tenant

Deploying across regions for low latency, disaster recovery, and compliance

PatternHowTrade-off
Active-PassivePrimary region serves traffic; standby for failoverSimple but standby idle; failover delay (minutes)
Active-ActiveBoth regions serve traffic; data replicatedLow latency globally but conflict resolution needed
Follow-the-SunRoute to region where it's business hoursGood for support/ops workloads
Guarantees: Multi-region provides disaster recovery (entire region can fail) and data residency compliance (GDPR: EU data stays in EU). Trade-off: cross-region replication lag (100-150ms RTT) and conflict resolution complexity.
Multi-Tenant: Shared infra (single DB, tenant_id column — cheap, noisy neighbor risk) vs Isolated (separate DB per tenant — expensive, strong isolation) vs Hybrid (shared compute, isolated DB). Salesforce uses shared multi-tenant. AWS provides isolated accounts.

Database Internals

How databases store and retrieve data — B-Trees, LSM Trees, WAL

StructureHowUsed ByOptimized For
B+TreeBalanced tree, sorted keys, data in leaves. O(log N).Postgres, MySQL InnoDBReads (point + range)
LSM TreeMemtable → flush to SSTables → compaction merges.Cassandra, RocksDB, LevelDBWrites (sequential I/O)
WALLog changes before applying. Replay on crash.Postgres, MySQL, etcd, KafkaDurability
Indexing: Clustered (data sorted by index, 1/table) · Non-clustered (pointer to data) · Composite (multi-column, leftmost prefix) · Covering (all query columns in index) · Partial (subset of rows). DB Locks: Row-level (InnoDB default, fine-grained) · Table-level (MyISAM, coarse) · Intent locks (signal intent to lock rows) · Advisory locks (app-level coordination).
Schema: Normalization (3NF, eliminate redundancy, needs JOINs) vs Denormalization (duplicate for read speed, no JOINs). ACID vs BASE: ACID = strong consistency (SQL). BASE = Basically Available, Soft state, Eventually consistent (NoSQL).

SQL (PostgreSQL, MySQL)

ACID transactions + complex queries. The default choice when you need consistency

Guarantees: Atomicity — all or nothing. Consistency — constraints always enforced. Isolation — MVCC (readers don't block writers). Durability — WAL survives crashes. These guarantees mean partial updates are impossible and committed data is never lost.
Postgres FeatureDetail
MVCCMulti-version — readers see snapshot, writers create new version. No read locks.
IndexesB-Tree (default), GIN (full-text/JSONB), GiST (geo), BRIN (large sequential)
JSONBBinary JSON with indexing — bridge SQL and document model
PartitioningRange/list/hash. Partition pruning speeds queries on large tables.
ExtensionsPostGIS (geo), TimescaleDB (time-series), Citus (distributed), pg_trgm (fuzzy)
Scaling: Read replicas (followers serve reads) · PgBouncer (connection pooling) · Citus/Vitess (sharding) · Vertical (bigger machine)
Limitations: Write bottleneck — single leader (~1-3K writes/sec). Resharding is painful — requires data migration and app changes. Rigid schema — ALTER on large tables can lock. Vertical scaling ceiling — biggest machine has limits.
Real-world: Instagram — sharded Postgres. Stripe — Postgres for payments. Supabase — "Firebase on Postgres".

NoSQL

Horizontal scaling, flexible schema, tunable consistency

TypeExamplesGuaranteeUse Case
Key-ValueRedis, DynamoDBO(1) lookup, partition tolerantSessions, cache
DocumentMongoDB, FirestoreSingle-doc ACID, flexible schemaProfiles, catalogs
Wide-ColumnCassandra, HBaseWrite-optimized, tunable consistencyTime-series, IoT
GraphNeo4j, NeptuneO(1) edge traversalSocial, fraud, recommendations
Cassandra Guarantees: Masterless ring — no SPOF. Tunable consistency — QUORUM (W+R>N = strong) or ONE (fast, eventual). Anti-entropy: Read repair, Merkle trees, hinted handoff keep replicas in sync.
Limitations: No complex joins — must denormalize or do multiple queries. No multi-row ACID — single-partition atomic only. Eventual consistency by default — stale reads possible. Must model around queries — bad data model = bad performance. Tombstone overhead — deletes create tombstones that slow reads until compaction.
Real-world: Discord — Cassandra (migrated to ScyllaDB). Netflix — Cassandra for viewing history. Uber — Cassandra for location data.

NewSQL

Global ACID at scale. Spanner (TrueTime + Paxos), CockroachDB (Raft), TiDB

Guarantees: Serializability globally — strongest isolation. SQL interface — familiar tools work. Trade-off: consensus latency (100ms+ multi-region) and 10x cost vs Postgres. Most apps don't need this.

Time-Series DBs

InfluxDB, Prometheus, TimescaleDB — optimized for append-heavy + range queries

Guarantees: High ingest (millions/sec). Auto-retention (TTL cleanup). Downsampling (1s→1min→1hr). Prometheus = pull-based scraping. TimescaleDB = Postgres extension (SQL for time-series).

Search (Elasticsearch)

Inverted index, BM25 ranking, fuzzy matching. NOT a primary data store

Guarantees: Sub-second full-text search across billions of docs. Shard resilience — replicas survive node failure. ELK Stack: Elasticsearch + Logstash + Kibana.
Limitations: Not ACID — eventually consistent (refresh interval delay before docs are searchable). Not a source of truth — must reindex from primary DB if data lost. Mapping explosion — too many dynamic fields break cluster. Deep pagination expensive — use search_after not from/size. JVM heap management — 50% of RAM, max 32GB.
Real-world: Wikipedia, GitHub code search, Uber trip search, Netflix log analysis.

Blob Storage (S3)

99.999999999% durability (11 nines). Cheap, infinite scale. Pre-signed URLs for direct access.

Guarantees: Strong read-after-write consistency (S3, since Dec 2020). Versioning protects against accidental deletes. Lifecycle policies auto-tier: Standard → IA → Glacier.
🗄️ 7 · CACHING

Caching Strategies

Store hot data in faster storage to reduce latency and DB load

StrategyFlowGuarantee
Cache-AsideApp checks cache → miss → query DB → write cacheCache only contains requested data. App controls logic.
Write-ThroughWrite to cache → cache syncs to DBAlways consistent. Write latency doubles.
Write-BehindWrite to cache → async flush to DBFast writes. Data loss risk if cache dies before flush.
Read-ThroughCache fetches from DB transparentlySimpler app code. Cache library dependency.
Invalidation: TTL (simple, stale until expiry) · Event-driven (CDC/app triggers delete, near real-time) · Version key (new version = auto miss). Eviction: LRU (most common) · LFU · FIFO.
Guarantees: Sub-millisecond reads for cached data (100x faster than DB). DB load reduction — cache absorbs read traffic spikes. Horizontal scalability — distributed caches (Redis Cluster, Memcached) scale linearly with nodes.
Limitations: Stale data — cache may serve outdated values until TTL expires or invalidation fires. Cache invalidation is hard — one of the "two hard problems in CS." Cold start — empty cache means all requests hit DB (thundering herd). Memory cost — RAM is expensive; can't cache everything. Consistency gap — cache and DB can diverge between writes and invalidation.
Thundering Herd: Cache expires → thousands hit DB simultaneously. Fix: mutex on cache miss (1 request fetches, others wait), probabilistic early expiration, stale-while-revalidate.
Real-world: Facebook uses Memcached (TAO) for social graph — billions of req/sec. Twitter caches timelines in Redis. Instagram caches explore feed. Target: cache hit rate >95%.

Redis

In-memory data store — sub-ms latency, 100K-1M ops/sec. Cache + data structures + messaging

StructureCommandsUse Case
StringSET/GET/INCR/SETNXCounters, session tokens, cache
HashHSET/HGET/HGETALLUser profiles, cache objects
ListLPUSH/RPOP/LRANGEJob queues, activity feeds
SetSADD/SMEMBERS/SINTERUnique visitors, tags, dedup
Sorted SetZADD/ZRANGE/ZRANKLeaderboards, rate limiting, priority queues
HyperLogLogPFADD/PFCOUNTApproximate unique count (0.81% error, 12KB)
GeoGEOADD/GEORADIUSNearby search, location tracking
Redis Pub/Sub: 
PUBLISH chat:room1 "Hello everyone!"
SUBSCRIBE chat:room1 → receives "Hello everyone!" (all current subscribers)
Subscriber joins later → NO message history (not stored)
Guarantees: Real-time delivery (<1ms latency). Fan-out — one publish → all subscribers receive simultaneously. Pattern matching — PSUBSCRIBE channel:* subscribes to all matching channels.
Limitations: No persistence — messages lost if no subscribers are listening at that moment. No replay — subscriber joining late misses all history. No acknowledgment — fire-and-forget, no delivery confirmation. No consumer groups — can't distribute work across consumers.
Real-world: Figma — real-time collaboration signals between users. Slack — online presence indicators. Cache invalidation across app servers (publish "invalidate:user:123" → all servers evict). Live sports scores fan-out.
Redis Streams: 
XADD orders * customer_id 5 amount 100 status "pending"  → Returns 1704067200000-0
XGROUP CREATE orders payment-service 0                     → Create consumer group
XREADGROUP GROUP payment-service consumer1 STREAMS orders > → Read unprocessed entries
XACK orders payment-service 1704067200000-0                → Acknowledge processing
Guarantees: Persistence — messages survive server restart (AOF/RDB). Replay — read from any offset with XRANGE. Consumer groups — track offset per group, auto-rebalance on consumer failure. Exactly-once semantics — with XACK, processed entries are marked done. Blocking reads — XREAD BLOCK 0 waits efficiently for new data.
Limitations: Single-threaded — can't parallelize per-entry processing within one stream (partition across multiple stream keys if needed). Not as fast as Kafka — single node <100K events/sec typical vs Kafka's millions. No ordering across streams — different stream keys = different order. Manual consumer management — no auto-scaling like Kafka consumer groups.
Real-world: Order processing pipelines — payment, fulfillment, analytics services each get independent consumer group on same stream. IoT sensor ingestion — append-only with TTL-based cleanup. Audit trails — immutable event log with replay capability. Good "lightweight Kafka" when you already have Redis and don't need Kafka's scale.
FeatureDetail
PersistenceRDB (periodic snapshot, fast restore, some data loss) · AOF (log every write, near-zero loss) · Hybrid (both, Redis 4.0+)
HASentinel — monitors master, auto-failover promotes replica
Cluster16,384 hash slots across N masters. Each master has replicas. Client-side routing.
RedlockDistributed lock: acquire on majority (N/2+1) nodes. TTL prevents deadlock.
Guarantees: Atomic operations (INCR, ZADD are thread-safe). Sub-ms latency (in-memory).
Limitations: RAM-bound — dataset must fit in memory ($$$). Single-threaded — one core bottleneck at extreme scale (use Redis Cluster to shard). Not a primary DB — not durable enough for source of truth. No complex queries — no SQL, no joins, no ad-hoc analytics. Replication is async — data loss possible on failover.
Real-world: Twitter — timeline cache. GitHub — job queues (Resque). Snapchat — rate limiting. Stripe — idempotency keys.

CDN

Cloudflare, CloudFront, Fastly — serve from edge PoPs globally. Pull (lazy) vs Push (proactive).

Guarantees: Low latency (<50ms from edge vs 150ms+ cross-ocean). DDoS absorption at edge. Origin offload — cache hit means origin never touched. Edge computing (Cloudflare Workers) runs logic at edge.
Limitations: Dynamic/personalized content harder to cache (user-specific responses). Cache invalidation complexity — purging stale content across 100s of PoPs takes time. Cost at high invalidation frequency. Debugging cache issues — hard to tell if user sees stale content.
Real-world: Netflix Open Connect — custom CDN in ISPs. Cloudflare serves 20%+ of web traffic.
📨 8 · MESSAGING

Message Queues (RabbitMQ / SQS)

Decouple producers from consumers — work distribution with retry + DLQ

Guarantees: At-least-once delivery (consumer must be idempotent). Ordering per queue (FIFO). Dead Letter Queue — after N retries, message moved to DLQ for inspection. Visibility timeout — message invisible to others while being processed.
Limitations: No message replay — messages deleted after consumption (unlike Kafka). Limited throughput for high-volume streaming (millions/sec). Fan-out to many consumers harder than Kafka (need exchange routing). No long-term retention — not an event store.
RabbitMQ exchanges: Direct (exact key) · Fanout (broadcast) · Topic (pattern: order.*.created) · Headers. SQS: Standard (at-least-once, best-effort order) vs FIFO (exactly-once, strict order, 3K msg/sec).

Apache Kafka

Distributed event streaming — 1M+ events/sec. Durable, replayable log.

Fan-out — fully managed, serverless. SNS+SQS pattern: SNS topic → multiple SQS queues (each with own retry/DLQ).

🔗 9 · CONSISTENCY & DISTRIBUTED SYSTEMS

CAP Theorem

Distributed system guarantees only 2 of 3: Consistency, Availability, Partition Tolerance

Since partitions are unavoidable, the real choice is CP vs AP during a partition. CP: Returns error rather than stale data (ZooKeeper, Spanner). AP: Returns possibly stale data (Cassandra, DynamoDB).
PACELC: If Partition → A or C. Else → Latency or Consistency. Most systems are tunable.

Concurrency Control

Multiple transactions accessing same data without corruption

Pessimistic: Lock first, assume conflict. SELECT ... FOR UPDATE. Pro Safe, no retry. Con Deadlock risk, bottleneck. Use: high contention (flash sales).
Optimistic: Assume no conflict, check version on write. UPDATE ... WHERE version=5. Pro High throughput, deadlock-free. Con Retry overhead. Use: low contention.
Isolation Levels: Read Uncommitted (dirty reads) → Read Committed (Postgres default) → Repeatable Read (MySQL default) → Serializable (strongest, slowest). Higher = fewer anomalies but more locking. MVCC (Postgres/MySQL) — readers see snapshot, no read locks. Deadlock prevention: Acquire locks in same order, use timeouts.

TypeHowUse Case

Replication

Copying data across nodes for availability and read scaling

TypeHowGuarantee
Leader-Follower1 writer, N readersSimple. Replication lag = stale reads on followers.

Consistent Hashing & Bounded Load

Hash ring that minimizes reshuffling when nodes are added/removed

How it works:
1. Hash space arranged as ring (0 to 2³²)
2. Nodes placed on ring by hash(nodeID)
3. Key assigned to first node clockwise from hash(key)
4. Add/remove node → only adjacent keys move (not all keys!)

Without consistent hashing: hash(key) % N → adding 1 node remaps ~ALL keys
With consistent hashing: adding 1 node remaps only ~1/N keys

Virtual Nodes (vnodes):
  Each physical node gets 100-256 positions on ring
  Why: prevents uneven distribution when few physical nodes
  Cassandra, DynamoDB, Redis Cluster all use vnodes

Bounded Load (Google, 2017):
  Problem: Even with vnodes, some nodes get more load (hash collisions)
  Fix: Cap each node at (1 + ε) × average load
  If node is "full", key goes to next node on ring
  Guarantee: No node gets more than (1+ε)× fair share
Guarantees: Minimal disruption — adding/removing nodes moves only O(K/N) keys. Even distribution with vnodes. Bounded load variant prevents hot nodes. Used by: Cassandra, DynamoDB, Redis Cluster, CDNs, Memcached.

Bloom Filters

Probabilistic membership test — "definitely not in set" or "probably in set"

How it works:
1. Bit array of size M, K hash functions
2. Insert "alice": hash1("alice")%M=23, hash2=45, hash3=78 → set bits [23,45,78]=1
3. Query "bob": hash1("bob")%M=23(1✓), hash2=60(0✗) → "bob" DEFINITELY NOT in set
4. Query "eve": all K bits are 1 → "eve" PROBABLY in set (false positive possible)

False positive rate = (1 - e^(-KN/M))^K
Typical: M = 10×N items, K ≈ 7 → ~1% false positive rate
Memory: ~100KB for 1 million items (vs storing actual items = MBs)

Key properties:
  ✓ No false negatives — if filter says "no", item is definitely absent
  ✗ False positives possible — if filter says "yes", item might not exist
  ✗ No deletion — can't safely unset bits (use Counting Bloom Filter instead)
  ✓ O(K) constant time for both insert and lookup
Guarantees: Zero false negatives — if Bloom filter says "not present", it's 100% correct. This is why it's used to avoid expensive disk reads: check filter first, only read disk if filter says "maybe yes".
Real-world: Cassandra/Bigtable — skip SSTables that definitely don't contain key (saves disk I/O). Chrome Safe Browsing — check URL against malware list without downloading entire list. Medium — avoid recommending already-read articles. CDNs — check if content is cached before origin fetch.

📊 13 · OBSERVABILITY

Logging

Structured JSON logs with correlation IDs. Levels: DEBUG → INFO → WARN → ERROR → FATAL

Guarantees: Immutability (forensics, compliance). Searchability (find all events for a user/request). Correlation IDs trace flow across services. Never log passwords/PII. Tools: ELK, Splunk, Loki, CloudWatch. Sample: 100% errors, 1-10% normal.

Metrics

Golden Signals (Google SRE): Latency · Traffic · Errors · Saturation

Guarantee: Metrics provide aggregate system health without individual event overhead. Track p50, p95, p99 (not averages — they hide tail latency). RED (Rate/Errors/Duration) for services. USE (Utilization/Saturation/Errors) for resources. Tools: Prometheus + Grafana (standard), Datadog.

Distributed Tracing

Track single request across services. Trace ID → Spans → Parent-child relationships → Latency breakdown.

Guarantees: Request correlation across all services. Bottleneck identification (which service is slow?). Dependency mapping (auto topology). Tools: OpenTelemetry (vendor-neutral standard), Jaeger, Zipkin, Datadog APM. Sample: 100% errors, 1-5% successes.

Monitoring & Alerting

Collect → Alert → Respond → Root Cause Analysis

Alert Rules: Only page if human action needed. Good: "error rate >1% for 2min". Bad: "any single error". SLO/SLI/SLA: SLI = metric (p99 latency). SLO = target (p99 <200ms, 99.9%). SLA = contract with consequences. Error Budget: 100% - SLO = budget for experimentation/risk. Tools: PagerDuty, Opsgenie, Grafana OnCall.
