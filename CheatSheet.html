Storage Systems

SQL — Relational Databases

PostgreSQL, MySQL, Cloud Spanner  —  ACID guarantees, complex queries, vertical scaling

Strengths
  • ACID transactions — all-or-nothing writes
  • Complex joins — no manual denormalization
  • Strong consistency — reads see latest write
  • Foreign key constraints enforce integrity
Weaknesses
  • Write bottleneck — ~3K writes/sec on Postgres
  • Resharding is painful
  • Rigid schema — ALTER on 100GB table = lock
  • Not built for massive unstructured data
Reach for SQL when
  • Financial systems — banking, payments, stocks
  • E-commerce with orders, inventory, transactions
  • Booking systems where double-booking is a disaster
  • Anything needing strong consistency + complex queries

The 4 ACID Promises

Atomicity — All or nothing. No half-baked writes.   Consistency — Constraints always hold.   Isolation — Concurrent transactions don't see each other's dirty work (MVCC).   Durability — Committed data survives crashes via WAL replay.

BEGIN;
SELECT * FROM inventory WHERE product_id=5 FOR UPDATE;  -- lock the row
UPDATE inventory SET quantity=9 WHERE product_id=5;
INSERT INTO orders (user_id, product_id, price) VALUES (123, 5, 50);
COMMIT;  -- either BOTH succeed or NEITHER does

NoSQL

DynamoDB, Cassandra, MongoDB, HBase  —  Horizontal scale, flexible schema, eventual consistency

Strengths
  • Horizontal scaling — add nodes = add capacity
  • No schema migration — add fields anytime
  • High write throughput — multi-master replication
  • Flexible — store anything, schema optional
Weaknesses
  • No complex joins
  • Eventual consistency — stale reads possible
  • No multi-doc ACID (single doc is atomic)
  • Must model data for access patterns upfront

NewSQL

Google Spanner, CockroachDB, TiDB  —  ACID at global scale. Best of both worlds — at a price.

Strengths
  • ACID globally — multi-region transactions
  • Familiar SQL interface, no retraining
  • Horizontal write scaling across regions
  • No eventual consistency surprises
Weaknesses
  • Consensus latency — 5–10ms single-region, 100ms+ multi-region
  • Cost premium — Spanner is ~10x more than Postgres
  • Overkill for single-region deployments
  • Requires distributed systems expertise

Hot take: Most apps don't need this. Sagas + Eventual Consistency is usually enough. Use CockroachDB (cheaper, Postgres-compatible) if you genuinely need NewSQL.

Time-Series Databases

InfluxDB, Prometheus, TimescaleDB, Druid  —  Optimized for time-stamped data at massive ingest rates

Strengths
  • High ingest — millions of events/sec
  • 10:1 compression — columnar + similar values
  • Auto-retention via TTL — no cleanup scripts
  • Lightning-fast range queries — ordered by time
Weaknesses
  • Limited query flexibility
  • Not for mutable data — updates/deletes are slow
  • No general-purpose queries
  • Cross-metric joins not supported
Reach for Time-Series when
  • Metrics and monitoring (Prometheus scraping services)
  • IoT sensors — temperature, pressure, location
  • APM — response times, error rates
  • Stock prices, OHLCV candles

Search Engines

Elasticsearch, OpenSearch, Solr  —  Full-text search at scale. Not a primary data store.

Strengths
  • Full-text search with stemming and tokenization
  • Fuzzy / typo-tolerant matching
  • Faceting, filtering, ranked results (BM25)
  • Autocomplete, aggregations
Weaknesses
  • Not ACID — no transactions
  • Eventual indexing — new docs not immediately searchable
  • Not a source of truth — must reindex from primary DB
  • Mapping explosion with too many unique fields

3 pitfalls that bite everyone

1. Mapping explosion — use dynamic: false.   2. Deep pagination — use search_after not from: 10000.   3. Hot shards — fix your distribution key.

JWT Structure

Header.Payload.Signature

Security Golden Rules

Store JWT in httpOnly cookie (not localStorage — XSS steals it instantly). Always HTTPS. Short-lived access tokens (15 min) + long-lived refresh tokens (7 days). Server-side blacklist on logout.

Caching

Cache Strategies

StrategyPatternUse When
Cache-AsideApp checks cache → miss → load DB → populate cacheRead-heavy, app controls logic
Write-ThroughWrite to cache → cache writes to DB synchronouslyCache + DB always in sync
Write-BehindWrite to cache → DB updated asyncHigh write throughput
Read-ThroughCache fetches from DB transparently on missCache acts as the data layer

Thundering Herd Problem

Cache entry expires at 12:00:00
1000 requests hit at 12:00:01 → ALL miss → ALL hit DB → DB falls over

Fixes:
  1. Probabilistic early expiration
  2. Mutex on cache miss
  3. Background refresh before TTL expires

Redis

In-memory Swiss Army knife — cache, message broker, data structure server. 100K–1M ops/sec.

Strengths
  • Sub-millisecond latency — 100x faster than a DB read
  • Rich data structures — Strings, Lists, Sets, ZSets, Hashes, Streams
  • Atomic operations — INCR, ZADD are thread-safe
  • Pub/Sub + Streams built in
Weaknesses
  • RAM-bound — memory is expensive
  • Not a primary DB
  • Single-threaded core
  • Data loss risk without AOF or RDB configured

CDN

Cloudflare, CloudFront, Fastly, Akamai  —  Bring content to users, not users to content

How it works

User in Mumbai → DNS → Nearest CDN PoP (Mumbai)
→ Cache hit? Return immediately (<10ms)
→ Cache miss? Fetch from origin (Oregon) → cache → return
→ Next user in Mumbai: instant cache hit

Use for: Images, video, JS/CSS bundles, static sites, HLS streaming segments.   Not for: Dynamic, personalized content (use edge compute for that).

Messaging

Message Queue

RabbitMQ, AWS SQS, Azure Service Bus  —  Decouple producers from consumers

Strengths
  • Decoupling — producer doesn't care about consumers
  • Work distribution across workers
  • Retry + Dead Letter Queue built in
  • At-least-once delivery guaranteed
Weaknesses
  • No replay — messages deleted after consumption
  • Not designed for millions/sec streaming
  • No native fan-out (use Pub/Sub)
  • No long-term retention

Use for: Order processing, email/SMS sending, background jobs, image resize pipelines, notification dispatch.

Apache Kafka

The distributed commit log. LinkedIn: 4+ trillion events/day.

Strengths
  • Extreme throughput — 1M+ events/sec per cluster
  • Durable + replayable — rewind to any point in time
  • Fan-out — multiple consumer groups read independently
  • Event sourcing — rebuild any state from the log
Weaknesses
  • Operational overhead — needs cluster management
  • Overkill for low volume
  • Ordering per partition only — not across partitions
  • At-least-once by default — idempotent consumers needed

Core Mental Model

Topic "orders"
├─ Partition 0 → Consumer Group A: Consumer 1
├─ Partition 1 → Consumer Group A: Consumer 2
└─ Partition 2 → Consumer Group A: Consumer 3
                ↕
Consumer Group B reads ALL partitions independently (different offset)

When to use Kafka vs alternatives

Use CaseKafkaRabbitMQ/SQSPub/Sub

APIs & Communication

REST

Stateless, HTTP-native, universally understood. The default choice.

Strengths
  • Universal — browser, mobile, IoT all speak it
  • HTTP caching built in (CDN, browser cache)
  • Simple semantics — GET/POST/PUT/DELETE
  • Stateless — easy to load-balance
Weaknesses
  • Over-fetching — all fields even if you need 3
  • Under-fetching — multiple requests for related data
  • Versioning tax — /v1/, /v2/ churn
  • N+1 queries require client-side aggregation

Pagination Patterns

?page=2&limit=10            → simple, but slow at high offsets
?offset=20&limit=10         → same problem
?cursor=abc123&limit=10     → cursor-based (best for infinite scroll)

gRPC

High-performance RPC over HTTP/2 with binary serialization. ~10x faster than REST.

Strengths
  • Protobuf — 3–5x smaller than JSON
  • Strong typing — .proto file is the contract
  • Bidirectional streaming
  • HTTP/2 multiplexing over one connection
Weaknesses
  • Browser hostile — can't call from browser directly
  • Harder to debug — no curl
  • Smaller ecosystem than REST
  • Overkill for simple CRUD

4 Call Types

1. Unary:            1 request → 1 response        (like REST)
2. Server streaming: 1 request → N responses       (stock feed)
3. Client streaming: N requests → 1 response       (batch upload)
4. Bidirectional:    N requests ↔ N responses      (chat, multiplayer)

Best for: Internal microservice-to-microservice calls, real-time bidirectional streams, performance-critical paths (10K+ RPS).

GraphQL

Client asks for exactly what it needs. No more, no less.

Strengths
  • No over/under-fetching — client shapes the response
  • Single endpoint — no versioning hell
  • Strongly typed schema — auto-docs, playground
  • Great developer experience via introspection
Weaknesses
  • Caching is hard — every query is unique
  • N+1 queries — naive resolvers destroy the DB
  • Query complexity attacks can DoS your backend
  • Per-field auth is complex to implement

N+1 Problem — and the Fix

// BAD: 1 query + 100 queries = 101 DB hits
User: { posts: (user) => db.query('SELECT * FROM posts WHERE user_id = ?', user.id) }

// GOOD: DataLoader batches → 2 total DB hits
const postLoader = new DataLoader(userIds =>
  db.query('SELECT * FROM posts WHERE user_id IN (?)', userIds)
);

Service Mesh

Istio, Linkerd, Consul Connect  —  Infrastructure layer for microservice networking

Envoy sidecar proxy runs alongside every service pod, intercepts all traffic. The service has no idea — it just sees plain HTTP. mTLS, retries, circuit breaking, and tracing are all handled transparently.

vs API Gateway: API Gateway handles North-South traffic (external → cluster). Service Mesh handles East-West traffic (service A → service B inside cluster). They complement each other.

Consistency & Distributed Systems

CAP Theorem

Pick 2 of 3. Since partition tolerance (P) is non-negotiable, you're really choosing CP vs AP.

TypeDuring a Partition...Examples
CPReturns error rather than stale dataHBase, Zookeeper, Spanner
APReturns possibly stale data, stays availableCassandra, DynamoDB, CouchDB

Real insight: CAP only applies during a network partition. The question is: what do you sacrifice when the network splits? PACELC is more useful in practice: during a Partition → A or C. Else (normal) → Latency or Consistency.

Concurrency Control

Pessimistic Locking — assume conflict, lock first

BEGIN;
SELECT * FROM inventory WHERE product_id = 5 FOR UPDATE; -- lock
UPDATE inventory SET quantity = quantity - 1 WHERE product_id = 5;
COMMIT; -- lock released
-- Safe. Slow under contention. Deadlock risk.

Optimistic Locking — assume no conflict, verify on write

-- Read with version
SELECT quantity, version FROM inventory WHERE product_id = 5; -- version=5
-- Update only if version unchanged
UPDATE inventory SET quantity = 9, version = 6
WHERE product_id = 5 AND version = 5;
-- 0 rows updated = conflict → retry
-- Fast, no blocking. Bad under high contention.

Simple rule: High contention (concert tickets) → Pessimistic. Low contention (user profile edits) → Optimistic.

Fault Tolerance & Reliability

Availability Math — memorize this

99.0%   → 87.6 hours downtime/year   (hobby project)
99.9%   → 8.76 hours downtime/year   (enterprise standard)
99.99%  → 52 minutes downtime/year   (high-availability)
99.999% → 5 minutes downtime/year    (Google/AWS-grade)

The 5 Resilience Patterns

PatternWhat it does
TimeoutsDon't wait forever — fail fast after N seconds
Retries + BackoffExponential delay between retries — let transient faults heal
Circuit BreakerStop hammering a broken service — fail fast while it recovers
BulkheadsIsolate thread pools — one bad service can't starve others
Health ChecksLiveness (is it alive?) + Readiness (can it serve traffic?)

Cardinal rule: No SPOFs. Always run at least 2 of everything (2 DBs, 2 LBs, 2 regions).

Sharding

Horizontal partitioning — split data across multiple DB nodes

StrategyHowWatch Out
Range shardingShard by value range (A–M, N–Z)Sequential writes create hot spots
Hash shardinghash(key) % NResharding requires remapping all keys
Directory shardingLookup table maps key → shardLookup table = single point of failure
Geo shardingShard by regionCross-region queries are expensive

Consistent Hashing

Hash ring 0 to 2³²
Nodes placed at hash(node_id) positions
Key assigned to first node clockwise from hash(key)
Add/remove a node → only adjacent keys move (minimal resharding)
Virtual nodes → each physical node = multiple ring positions (even distribution)

Quorum Math

N = total replicas, W = write quorum, R = read quorum
W + R > N → Strong consistency guaranteed
N=3, W=2, R=2 → Strong (classic config)
N=3, W=1, R=1 → Fast + high availability, eventual consistency

Follow a single request across every service it touched

  │   ├── DB Query     span_id=1.1 (10ms)

Design Patterns

Bloom Filters

Probabilistic membership test — "is this item definitely NOT in the set?" — in ~1.25MB

Memory Win

1M URLs in a Hash Set:     ~125 MB
1M URLs in a Bloom Filter: ~1.25 MB  (100x smaller, ~1% false positive rate)
Advantages
  • Tiny memory footprint
  • O(K) constant-time lookup
  • No false negatives (if bit not set, item is definitely absent)
Limitations
  • False positives (tunable, not eliminatable)
  • Can't delete items (use Counting Bloom Filter)
  • False positive rate grows as filter fills

In the wild: Cassandra/Bigtable (skip SSTables on disk), malware URL detection, cache miss prevention.

Circuit Breaker

Fail fast. Don't cascade. Heal automatically.

[CLOSED] → requests pass through normally
    ↓ (5 failures)
[OPEN]   → requests fail immediately (don't call broken service)
    ↓ (wait 30s)
[HALF-OPEN] → let 1 test request through
    ├── success → [CLOSED]
    └── failure → [OPEN]

Without vs With Circuit Breaker

Without: Payment service down → Order service calls it (5s timeout)
         1000 req/s × 5s = 5000 stuck threads → Order service dies → cascade

With:    5 timeouts → circuit opens → requests 6-∞ fail in <1ms
         Order service stays healthy → Payment service recovers quietly

↑ Back to SQL

SQL — Deep Dive

B-Tree Indexes

Almost all SQL indexes are B-Trees. Index columns used in WHERE, JOIN, ORDER BY. Put the most selective column first in composite indexes. Too many indexes = slow writes. Use partial indexes (WHERE status = 'active') to index subsets.

Write-Ahead Log (WAL)

Every write is first logged to WAL (sequential disk write = fast). Data pages updated asynchronously. On crash: replay WAL to restore committed state. This is why COMMIT is durable even if the OS crashes mid-write.

Scaling SQL

1. Vertical         → Bigger machine. Easiest, limited ceiling.
2. Read replicas    → Async replication to followers. Reads scale, writes don't.
3. Connection pool  → PgBouncer/ProxySQL. DB connections are expensive — reuse them.
4. Sharding         → Partition by shard key. Complex queries suffer.

NoSQL — Deep Dive

Cassandra Data Modeling Mantra

"Model around your queries, not your data." Table per query. If you need 3 access patterns, build 3 tables. Data duplication is acceptable — disk is cheap, joins aren't.

Cassandra Consistency Tuning

QUORUM write + QUORUM read: W+R > N → Strong consistency
ONE write + ONE read:       Fastest, eventual consistency
ALL write + ONE read:       Durable writes, but write latency = slowest replica

DynamoDB Access Patterns

Primary key = partition key + optional sort key. GSI (Global Secondary Index) = alternate access patterns without changing table design. LSI (Local Secondary Index) = different sort key on same partition key.

↑ Back to TCP/UDP

TCP/UDP — Deep Dive

TCP Congestion Control

Slow start: exponentially increase from 1 MSS. Congestion avoidance: linear increase after threshold. Fast retransmit: 3 duplicate ACKs = infer loss → retransmit immediately. CUBIC: better for high-bandwidth long-distance links.

Why UDP for DNS

DNS query fits in 1 packet (~50 bytes). TCP handshake = 1.5 RTT overhead before the query even sends. UDP: send query, get response = 0.5 RTT. If packet lost? Just send another — much faster than TCP retry.

Why UDP for Gaming

A lost packet = old position data (irrelevant in 16ms). TCP would stall waiting for retransmission. UDP: just process the next packet — older position data is stale anyway.

CAP Theorem — Deep Dive

Why P is Non-Negotiable

In any distributed system across multiple machines: network cables can be cut, switches can fail, cloud regions can be isolated, BGP misconfigurations happen. If you don't tolerate partitions, you have a single-node system. So the real choice: during a partition, CP or AP?

The Consistency Spectrum

↑ Back to Concurrency

Concurrency Control — Deep Dive

Deadlock — How It Happens and How to Prevent It

Transaction A:               Transaction B:
Lock row user_id=1           Lock row user_id=2
Try lock user_id=2 → wait    Try lock user_id=1 → wait
           DEADLOCK! Both waiting forever.

Prevention:
1. Lock ordering: always acquire in same order (user_id ASC)
2. Lock timeouts: after 100ms waiting, abort and retry
3. Optimistic locking: no locks, no deadlocks

MVCC (Multi-Version Concurrency Control)

Every row has xmin (created by txn) and xmax (deleted by txn). Reader sees a snapshot at transaction start time. Writer creates new row version — doesn't overwrite. Readers never block writers. Writers never block readers. This is how Postgres achieves ACID with high concurrency.

↑ Back to Sharding

Sharding — Deep Dive

Resharding with Zero Downtime (Double-Shard Strategy)

1. Create 8 new shards (shadow)
2. Dual-write: all new writes go to both old (4) and new (8) shards
3. Backfill: copy existing data from old → new shards
4. Verify: data in 8 shards matches 4 shards
5. Cut over reads to new 8 shards
6. Stop writes to old 4 shards, then delete them
Result: Zero downtime, minimal data movement

Celebrity / Hotspot Mitigation

Problem: user_id=beyonce (300M followers) → all reads hit one shard

Solution 1 — Key salting:
  Write to: "user:beyonce:1", ..., "user:beyonce:100"
  Read from: any of the 100 keys (or all 100 and sum)

Solution 2 — Dedicated shard:
  celebrity users → shard 0 (extra capacity)
  regular users   → shards 1-N (hash-based)

Tuning for Target False Positive Rate

Circuit Breaker — Deep Dive

Production Implementation (Python)

class CircuitBreaker:
    def __init__(self, failure_threshold=5, timeout=30):
        self.state = "CLOSED"
        self.failure_count = 0
        self.last_failure_time = None

    def call(self, func, *args, **kwargs):
        if self.state == "OPEN":
            if time.time() - self.last_failure_time > self.timeout:
                self.state = "HALF_OPEN"
            else:
                raise CircuitOpenError("Fast failing")
        try:
            result = func(*args, **kwargs)
            self._on_success()
            return result
        except Exception as e:
            self._on_failure()
            raise

    def _on_failure(self):
        self.failure_count += 1
        self.last_failure_time = time.time()
        if self.failure_count >= self.failure_threshold:
            self.state = "OPEN"

Fallback Strategies When Circuit is Open

try:
    return payment_service.charge(amount)
except CircuitOpenError:
    if is_idempotent_retry(request):
        return queue_for_async_retry(request)   # retry later
    elif has_cached_result(request):
        return get_cached_result(request)        # serve stale
    else:
        return ServiceUnavailableResponse()      # fail gracefully

Blob Storage — Deep Dive

Pre-signed URLs — the Right Pattern for Uploads

Without pre-signed URLs:
  Client → Your server (entire 500MB file) → S3
  Your server is a bottleneck + pays bandwidth cost

With pre-signed URLs:
  Client → Your server: "I want to upload a 500MB file"
  Your server → Client: pre-signed URL (valid 15 min)
  Client → S3 directly (no server involvement, no bandwidth cost)

url = s3.generate_presigned_url('put_object',
  Params={'Bucket': 'my-bucket', 'Key': f'uploads/{user_id}/{filename}'},
  ExpiresIn=900
)

Multipart Upload for Large Files

File: 5GB video
→ Split into 100 parts (50MB each)
→ Upload parts in parallel
→ S3 assembles parts server-side
Benefits: resume after failure, parallel = faster, required for files > 5GB

DNS — Deep Dive

Resolution in Detail

You type: www.example.com
1. Browser DNS cache: miss
2. OS DNS cache: miss
3. Recursive resolver asks Root NS: "Who handles .com?"
   → Root NS: "Ask 192.5.6.30 (Verisign .com TLD)"
4. Asks .com TLD: "Who handles example.com?"
   → .com TLD: "Ask 205.251.196.1 (authoritative)"
5. Asks authoritative NS: "What's www.example.com?"
   → Returns: 93.184.216.34
Total: ~50–100ms cold. <1ms cached.

GeoDNS Routing

Client in India   → region = Asia   → returns Singapore IP
Client in Germany → region = Europe  → returns Frankfurt IP
Same domain, different IPs, each routed to nearest region.
Used by: Netflix, Google, every major CDN.

Data Warehouse — Deep Dive

Partition Pruning

Table partitioned by month (2022-01 to 2026-04)
Query: "SELECT * FROM orders WHERE month = '2026-04'"
→ BigQuery reads ONLY April 2026 partition
→ Skips 51 other months → 50x less data → 50x cheaper

Cluster by user_id within partition:
→ "WHERE user_id = 123" → reads only that user's block → another 10x improvement

ELT vs ETL

ETL (traditional): Extract → Transform (Python/Spark) → Load clean data
  Problem: Transform step is bottleneck, hard to scale.

ELT (modern): Extract → Load raw → Transform IN the warehouse (SQL)
  BigQuery/Snowflake are powerful enough to transform in SQL.
  dbt (Data Build Tool) = write SQL transforms, version control, test, deploy.
