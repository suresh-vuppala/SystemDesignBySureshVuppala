How does Slack handle 10+ billion messages per day across the globe?

Scale: 10+ billion messages/day | Requirements: Message ordering, channel fanout, searchability, near real-time delivery across multiple regions
Key Topics:
WebSockets Kafka Fanout Service Presence Service Elasticsearch Multi-region Replication Offline Delivery Event Streaming Redis

Architecture Solution

Real-Time Path: Client → WebSocket Gateway → Message Service → Kafka (persistence) → Redis Pub/Sub (fanout) → Connected clients
  • Ingestion: WebSocket gateways accept messages, write to Kafka for durability and ordering
  • Fanout: Kafka → Fanout Service → Redis Pub/Sub channels by channel_id for instant delivery to connected users
  • Offline Delivery: Messages queued in database for offline users, delivered on reconnection
  • Search: Kafka → Elasticsearch pipeline for instant full-text search across billions of messages
  • Ordering: Partition messages by channel_id in Kafka, process in order per partition
  • Multi-region: Primary datacenter ingests all messages, replicate to secondary regions for read replicas and failover

How does WhatsApp deliver messages to billions of users even when they're offline?

Scale: Billions of users | Requirements: Reliable offline delivery, read receipts, typing indicators, push notifications, end-to-end encryption
Key Topics:
WebSockets Push Gateway Retry Queue Encryption Presence Cassandra Message ACKs Fanout

Architecture Solution

Delivery Guarantee: Message → Queue → Recipient check (online/offline) → if offline: persist + push notification
  • Message Storage: Cassandra (write-heavy, distributed) for durable message history per user
  • ACK System: Three-state ACKs (sent, delivered, read) via WebSocket or push notification
  • Offline Delivery: Redis queue per user; retry up to 30 days before discarding
  • Push Notifications: APNS/FCM gateway handles retries and device token management
  • Typing Indicators: Ephemeral WebSocket events (no persistence), immediate fanout to conversation participants
  • Encryption: End-to-end keys managed client-side; server never sees plaintext

How does Discord handle millions of concurrent WebSocket connections without crashing?

Scale: Millions of concurrent WebSocket connections | Requirements: Real-time messaging, presence updates, voice signaling, minimal latency
Key Topics:
Gateway Servers WebSockets Sharding Heartbeats Presence Tracking Event Routing Voice Signaling

Architecture Solution

Sharding Strategy: Partition users by shard ID to distribute millions of connections across multiple gateway servers
  • Sharding: Hash user_id mod N shards; each gateway owns subset of connections; clients know shard assignment
  • Inter-shard communication: Redis Pub/Sub for events crossing shard boundaries (e.g., send message in guild visible to multiple shards)
  • Heartbeats: Client sends heartbeat every 45s; server responds; missing 2 heartbeats = timeout/reconnect
  • Presence: On connect: publish to Redis; on disconnect: clear; subscribers notified instantly
  • Voice Signaling: SDP offer/answer exchanged via WebSocket before P2P connection established
  • Failover: Other gateways detect connection loss, clients auto-reconnect to healthy gateway

How do chat apps guarantee message ordering when entire regions go down?

Scale: Multi-region deployment | Challenge: Guarantee message ordering within conversations during regional failures and network partitions
Key Topics:
Kafka Partitioning Vector Clocks Lamport Timestamps Consensus Replication Eventual Consistency

Architecture Solution

Ordering Guarantee: Use Lamport timestamps + Kafka partitioning to ensure total ordering within a conversation
  • Lamport Timestamp: Each message gets (logical_clock, server_id); increment on each message, take max(local, received+1)
  • Kafka Partitioning: Partition by conversation_id; all messages in conversation go to same partition → strict ordering
  • Multi-region: Primary region (US) receives writes first, generates Lamport timestamp; replicate to secondaries (EU, APAC)
  • Failover: If primary fails, secondary can become primary; Lamport clock ensures no duplicate or reordered messages
  • Write Handling: During partition: accept writes locally (queue in Kafka), use vector clocks for causality; merge on heal

How do Telegram groups with 1 million members avoid message delivery meltdown?

Scale: 1 million members in single group | Challenge: Prevent fanout bottlenecks and message delivery overload
Key Topics:
Fanout Optimization Push/Pull Hybrid Partitioning Pub/Sub Batch Delivery

Architecture Solution

Hybrid Push/Pull Model: Large groups use pull-based delivery to prevent fanout explosion
  • Push Model: For groups < 10K members, use Redis Pub/Sub for instant delivery (fanout acceptable)
  • Pull Model: For groups > 10K members, write message to database; clients poll/long-poll for new messages
  • Hybrid: Online users (tracked in Redis) receive push instantly; offline users pull on reconnect
  • Partitioning: Shard the 1M members across 100 partition groups; each partition has <10K connected clients
  • Batch Delivery: Instead of 1M individual fanout messages, batch and send to partition leaders
  • Caching: Last 10K messages cached in Redis; new users (pull) fetch from cache + database

How does WhatsApp show "online" status for 100M+ users instantly?

Scale: 100M+ users | Challenge: Detect online/offline status changes in real-time without overloading Redis or gateways
Key Topics:
Heartbeats Redis TTL Presence Aggregation WebSockets State Sync

Architecture Solution

Presence Pipeline: Heartbeat → Redis TTL → Presence Cache → Event Fanout (only to interested users)
  • Heartbeat: Client sends heartbeat every 30s; server updates Redis key with TTL 60s (grace period)
  • Presence Cache: In-memory cache per gateway of connected users' presence status (last seen, device, status)
  • Fanout Strategy: Only send presence updates to users in contact list (not all 100M users globally)
  • Batch Updates: Collect presence changes for 1s, then batch send (1000 status updates in single message)
  • Sharding: Hash (user_id, contact_id) to partition presence subscriptions across multiple presence services
  • TTL Expiry: Background job periodically scan expired keys, emit offline event to interested subscribers

How does Slack show "typing..." indicators for millions of users without storing anything?

Scale: Millions of ephemeral events/second | Challenge: Handle real-time typing events without persisting to database
Key Topics:
Ephemeral Events WebSockets In-memory Messaging Redis Pub/Sub

Architecture Solution

No Persistence: Typing indicators flow through WebSocket/Redis Pub/Sub, never touch database
  • Client Event: User types → client sends WebSocket event: {type: 'typing', conversation_id, user_id}
  • Fanout: Gateway publishes to Redis channel: typing:{conversation_id} (no queue, instant Pub/Sub)
  • Delivery: Other connected clients in conversation subscribed to same channel receive typing event instantly
  • Debounce: Client displays "user is typing..." for 3s; if no new event, hide (server doesn't need to send "stop typing")
  • Cleanup: No TTL needed; typing event naturally expires when user stops/disconnects
  • Scalability: Redis Pub/Sub overhead minimal (~100 bytes per event); can handle 10M events/sec on single Redis instance

How does Slack search through billions of messages in milliseconds?

Scale: Billions of messages | Challenge: Support instant search with near real-time indexing (< 5s index latency)
Key Topics:
Elasticsearch Inverted Index Kafka Streams Indexing Pipeline Storage Optimization

Architecture Solution

Real-Time Indexing Pipeline: Kafka → Processing → Elasticsearch bulk indexing every 5s
  • Kafka Source: All messages written to Kafka topic; indexing consumes in real-time
  • Enrichment: Add conversation_id, sender metadata, timestamps; optionally strip large binary payloads
  • Bulk Indexing: Batch 10K documents, send bulk index every 5s (balance freshness vs overhead)
  • Inverted Index: Elasticsearch tokenizes message text, creates index for instant full-text search
  • Sharding: Shard Elasticsearch by conversation_id or date (old conversations rarely searched; move to cold storage)
  • Query Path: User searches → Elasticsearch query → ranked results (relevance + recency) → return top 100
  • Storage: Compress Elasticsearch indices; delete/archive indices > 1 year old

How does WhatsApp ensure you never get duplicate push notifications?

Scale: Millions of devices | Requirements: Handle delayed delivery, device reconnection, notification deduplication, retry without duplicates
Key Topics:
APNS / FCM Retry Queues Idempotency Notification Fanout Device Management

Architecture Solution

Deduplication + Retry: Use idempotency keys to prevent duplicate push notifications on retries
  • Push Event: Create push notification with idempotency_key = sha256(user_id + message_id + device_id)
  • Retry Queue: Kafka topic for push notifications; consumer sends to APNS/FCM with idempotency key
  • APNS/FCM Handling: If device invalid/unregistered, remove device token from user profile
  • Exponential Backoff: Retry 1s, 5s, 30s, 5min, 1h; give up after 24h (user likely switched devices)
  • Deduplication: Maintain set of delivered idempotency keys (Redis, 24h TTL); skip re-sending if already sent
  • Silent Failures: Track delivery metrics (success rate, latency); alert if FCM success rate drops below 95%

How does Telegram keep your messages perfectly synced across phone, tablet, and desktop?

Scale: Multiple devices per user | Challenge: Keep messages, media, read status consistent across phones, tablets, desktop without conflicts
Key Topics:
Event Sourcing Sync Tokens Device Sessions Conflict Resolution State Versioning

Architecture Solution

Event Sourcing + Sync Tokens: Track all state changes as immutable events; devices sync using tokens
  • State Events: All actions (send message, mark read, delete, edit) stored as events with timestamp and device_id
  • Sync Tokens: Each device maintains last_sync_token; server returns delta (all events since token)
  • Conflict Resolution: Last-write-wins by timestamp; all devices apply events in chronological order
  • Device Sessions: Track active sessions per user; on login from new device, include full state snapshot
  • Offline Support: Device queues events locally; on reconnect, send to server, receive acks, merge with server state
  • Storage: Event log immutable; compute current state by replaying events (CQRS pattern)
