🔗 9 · CONSISTENCY & DISTRIBUTED SYSTEMS

CAP Theorem

Distributed system guarantees only 2 of 3: Consistency, Availability, Partition Tolerance

Since partitions are unavoidable, the real choice is CP vs AP during a partition. CP: Returns error rather than stale data (ZooKeeper, Spanner). AP: Returns possibly stale data (Cassandra, DynamoDB).
PACELC: If Partition → A or C. Else → Latency or Consistency. Most systems are tunable.

Guarantees: Safety — all nodes agree. Liveness — eventually makes progress. Survives f failures with 2f+1 nodes. FLP Impossibility: No deterministic consensus in async system with even 1 crash (practical systems use timeouts).

Concurrency Control

Multiple transactions accessing same data without corruption

Pessimistic: Lock first, assume conflict. SELECT ... FOR UPDATE. Pro Safe, no retry. Con Deadlock risk, bottleneck. Use: high contention (flash sales).
Optimistic: Assume no conflict, check version on write. UPDATE ... WHERE version=5. Pro High throughput, deadlock-free. Con Retry overhead. Use: low contention.
Isolation Levels: Read Uncommitted → Read Committed (Postgres default) → Repeatable Read (MySQL default) → Serializable (strongest, slowest). MVCC — readers see snapshot, no read locks. Deadlock prevention: Acquire locks in same order, use timeouts.
