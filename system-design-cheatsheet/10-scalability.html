Partitioning

Splitting data across nodes — the foundation of horizontal scaling

TypeHowUse Case
Horizontal (Row)Split rows across nodes by partition keyMost common. Each node holds subset of rows.
Vertical (Column)Split columns across tables/servicesSeparate hot columns from cold. Microservice decomposition.
RangePartition by value range (A-M, N-Z, dates)Efficient range queries. Risk: hot spots.
Hashhash(key) % NEven distribution. Resharding requires remapping.
ListExplicit mapping (region=US → shard 1)Geo-partitioning, compliance.
Guarantees: Partitioning provides linear write scaling. Partition pruning skips irrelevant partitions. Cross-partition queries are expensive (scatter-gather).
Hotspot Mitigation: Celebrity problem — fix: add random suffix → scatter across shards → aggregate on read.

Replication

Copying data across nodes for availability and read scaling

TypeHowGuarantee
Leader-Follower1 writer, N readersSimple. Replication lag = stale reads on followers.
Multi-LeaderMultiple writers, conflict resolutionMulti-region writes. Conflict resolution needed (LWW, CRDTs).
LeaderlessAny node accepts writes (quorum)High availability. Tunable consistency (W+R>N).
Quorum Guarantee: N=3 replicas. W=2, R=2 → W+R=4 > N=3 → strong consistency. W=1, R=1 → fast but eventual.
Sync vs Async Replication:

Synchronous: 
User → API Gateway → Master → Replica → ACK → Primary → Response to User
✔ Strong consistency | ✔ Zero replication lag | ✖ Slower writes

Asynchronous: 
User → API Gateway → Master → Response to User
                        │
                        └──→ Replica (background replication)
✔ Faster writes | ✖ Eventual consistency | ✖ Data loss risk

Sharding

Horizontal partitioning across independent database instances

Sharding vs Partitioning: Partitioning = splitting within one DB. Sharding = splitting across multiple independent DB instances.
StrategyGuaranteeRisk
Hash ShardingEven distribution across shardsResharding requires data migration
Range ShardingEfficient range queries within shardHot spots for sequential keys
Directory ShardingFlexible mapping via lookup tableLookup table = SPOF
Geo ShardingData locality + complianceCross-region queries expensive
Real-world: Instagram shards Postgres by user_id. Vitess (YouTube) — MySQL sharding middleware. Citus — distributed Postgres.

2 Consistent Hashing — The Basic Idea
  1. Hash servers onto a ring (0 to 2³²)
  2. Hash keys onto the same ring
  3. Each key goes to the first server clockwise

Key → Server Mapping

K1→S1 K2→S1 K3→S2 K4→S3 K5→S3 K6→S1
S1 owns range (225, 75]
S2 owns range (75, 150]
S3 owns range (150, 225]
Only keys in a server's range belong to that server
3 What Happens When the Cluster Changes?

Add S4 at position 200

K5 was on S3 → now on S4
All other keys unchanged
Only keys in (150, 200] move from S3 → S4

Remove S2

K3 was on S2 → now on S3 (next clockwise)
All other keys unchanged
Keys in (75, 150] move to next server clockwise

5 How the Ring is Built (High Level)
  1. Discover servers in cluster — get the list of active nodes
  2. Generate virtual nodes — hash(serverId + i) for i = 1..N
  3. Sort points on the ring — keep a sorted structure (e.g., array / skip list)
  4. Locate key — hash(key) → find first node clockwise (binary search)
  5. Handle replicas (optional) — pick next R distinct physical nodes on the ring for replication

✓ Virtual nodes essential for even distribution
✓ Solves placement stability, not hot keys

⚠ Watch Out For

⚠ Hot keys can still overwhelm a node
⚠ Data size imbalance (not just key count)
⚠ Different ring views → inconsistent routing
⚠ Too few virtual nodes → uneven load
⚠ Rebalancing is not free — do it intentionally
Mental Model: Consistent hashing is not about hashing. It's about stable key placement under change.

Bloom Filters

Probabilistic membership test — "definitely not in set" or "probably in set"

How it works:
1. Bit array of size M, K hash functions
2. Insert "alice": hash1("alice")%M=23, hash2=45, hash3=78 → set bits [23,45,78]=1
3. Query "bob": hash1("bob")%M=23(1✓), hash2=60(0✗) → "bob" DEFINITELY NOT in set
4. Query "eve": all K bits are 1 → "eve" PROBABLY in set (false positive possible)

False positive rate = (1 - e^(-KN/M))^K
Typical: M = 10×N items, K ≈ 7 → ~1% false positive rate
Memory: ~100KB for 1 million items

Key properties:
  ✓ No false negatives — if filter says "no", item is definitely absent
  ✗ False positives possible — if filter says "yes", item might not exist
  ✗ No deletion — use Counting Bloom Filter instead
  ✓ O(K) constant time for both insert and lookup
Guarantees: Zero false negatives. Used to avoid expensive disk reads: check filter first, only read disk if filter says "maybe yes".
Real-world: Cassandra/Bigtable — skip SSTables. Chrome Safe Browsing — check URL against malware list. Medium — avoid recommending already-read articles. CDNs — check if content is cached.

Rate Limiting

Throttle requests to protect services from abuse

AlgorithmHowGuarantee
Token BucketTokens added at fixed rate; request consumes 1Allows bursts up to bucket size. Most common.
Sliding WindowWeighted blend of current + previous windowAccurate + memory efficient. No edge bursts.
Leaky BucketFixed output rate; excess queued/droppedSmooth output. No burst tolerance.
Real-world: Stripe — 100 req/sec per API key. GitHub — 5000 req/hour. Apply at: API Gateway (preferred), reverse proxy, or Redis (INCR+TTL).
