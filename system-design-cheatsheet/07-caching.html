🗄️ 7 · CACHING

Caching Strategies

Store hot data in faster storage to reduce latency and DB load

StrategyFlowGuarantee
Cache-AsideApp checks cache → miss → query DB → write cacheCache only contains requested data. App controls logic.
Write-ThroughWrite to cache → cache syncs to DBAlways consistent. Write latency doubles.
Write-BehindWrite to cache → async flush to DBFast writes. Data loss risk if cache dies before flush.
Read-ThroughCache fetches from DB transparentlySimpler app code. Cache library dependency.
Invalidation: TTL (simple, stale until expiry) · Event-driven (CDC/app triggers delete, near real-time) · Version key (new version = auto miss). Eviction: LRU (most common) · LFU · FIFO.
Guarantees: Sub-millisecond reads for cached data (100x faster than DB). DB load reduction. Horizontal scalability — distributed caches scale linearly with nodes.
Limitations: Stale data. Cache invalidation is hard. Cold start — empty cache means all requests hit DB (thundering herd). Memory cost. Consistency gap.
Thundering Herd: Cache expires → thousands hit DB simultaneously. Fix: mutex on cache miss, probabilistic early expiration, stale-while-revalidate.
Real-world: Facebook uses Memcached (TAO) for social graph. Twitter caches timelines in Redis. Instagram caches explore feed. Target: cache hit rate >95%.

Why Redis is So Fast?

Multiple design decisions compound together

In-Memory — all data in RAM. RAM ~100ns vs disk ~10ms = 100,000x faster.
Single-Threaded Core — no locks, no context switching, no race conditions. Commands execute atomically.
Non-Blocking I/O — epoll multiplexing — single thread watches 100K+ sockets.
Efficient Internals — SDS strings, ziplist, skiplist — CPU cache-friendly.
Simple Protocol (RESP) — plain text, O(1) parsing.
Pipelining — batch multiple commands in one TCP round trip — up to 10x throughput gain.
No Query Planner — commands are direct operations — no SQL parsing, no optimizer.
RAM ~100ns · SSD ~100µs · HDD ~10ms
Single GET/SET  → 100K–1M ops/sec
With Pipelining → up to 10x gain
Bottleneck = NETWORK, not CPU → single thread is enough
Redis 6.0+: I/O threads for network read/write — still single-threaded for command execution.
Watch out: RAM-bound · avoid KEYS * / SMEMBERS on huge sets · use Redis Cluster to shard.

🗄️ Redis as Cache

Pattern: App checks Redis first → cache hit returns instantly · cache miss → fetch from DB → write to Redis with TTL → serve.
Strategies: Cache-aside (most common). Write-through. Write-back. TTL-based eviction.
Eviction Policies: LRU (default for cache). LFU. allkeys-lru. volatile-lru.

📢 Redis Pub/Sub

PUBLISH  chat:room1 "Hello!"           → sends to all current subscribers
SUBSCRIBE chat:room1                   → receives "Hello!" instantly
PSUBSCRIBE chat:*                      → pattern match — all chat channels
Subscriber joins later → NO history   → messages already gone
Guarantees: Real-time delivery (<1ms). Fan-out. Pattern matching.
Limitations: No persistence. No replay. No acknowledgment. No consumer groups.
Real-world: Figma — real-time collaboration. Slack — online presence. Cache invalidation across servers.

🌊 Redis Streams

XADD   orders * customer_id 5 amount 100 status "pending"  → 1704067200000-0
XGROUP CREATE orders payment-service 0                      → create consumer group
XREADGROUP GROUP payment-service consumer1 STREAMS orders > → read unprocessed
XACK   orders payment-service 1704067200000-0               → acknowledge done
XRANGE orders - +                                           → replay all entries
Guarantees: Persistence. Replay. Consumer groups. At-least-once. Blocking reads.
Limitations: Single node — <100K events/sec. RAM-bound. Good lightweight Kafka when you already have Redis.

🔧 Persistence, HA & Cluster

FeatureDetail
PersistenceRDB (periodic snapshot) · AOF (log every write) · Hybrid (both, Redis 4.0+)
SentinelHA mode — monitors master, auto-failover promotes replica. No sharding.
Cluster16,384 hash slots across N masters. Each master has replicas. Horizontal scale.
RedlockDistributed lock — acquire on majority (N/2+1) nodes. TTL prevents deadlock.
Guarantees: Atomic operations. Sub-ms latency. Lua scripting — multi-step atomic transactions.
Limitations: RAM-bound. Single-threaded core. Not a primary DB. No complex queries.

CDN

Cloudflare, CloudFront, Fastly — serve from edge PoPs globally. Pull (lazy) vs Push (proactive).

Guarantees: Low latency (<50ms from edge). DDoS absorption at edge. Origin offload. Edge computing (Cloudflare Workers) runs logic at edge.
Limitations: Dynamic/personalized content harder to cache. Cache invalidation complexity. Cost at high invalidation frequency.
Real-world: Netflix Open Connect — custom CDN in ISPs. Cloudflare serves 20%+ of web traffic.
