Architecture Overview

Write path (send) → Kafka → Fan-out workers → Gateway → WebSocket push to clients

Fan-out Strategy — The Hard Part

50K-member channel: can't write 50K times per message. Push only to online members (~10% of total)

Channel SizeStrategyWhy
<100 membersWrite-time fan-out — push to each member's Gateway immediatelyLow latency, simple, small fan-out
100–50K membersHybrid — push to online members only, skip offline50K members but only 5K online → fan-out to 5K not 50K
>50K membersRead-time fan-out — write to channel log, clients stream from cursorWrite amplification too expensive at this scale
Presence tracking: channel:{id}:online → Redis SET (updated on WS connect/disconnect). Fan-out worker iterates this set, not full membership. For 50K-member channel with 5K online → 90% less work.

Ordering & Consistency

Users must see messages in the same order — Kafka partition per channel + sequence numbers

How Ordering Works

1. All messages for #general go to same Kafka partition (key=channel_id)
2. Single consumer per partition → messages processed in order
3. Each message gets channel_seq (monotonic increment per channel)
4. Client tracks last received seq. If gap detected → request backfill
5. On reconnect: client sends last_seq=47 → server returns msgs 48+

What Can Go Wrong

WS drops: Client reconnects, sends last_seq → backfill from Vitess
Gateway crash: All users on that gateway reconnect to another → Redis mapping updated
Duplicate delivery: Client deduplicates by channel_seq (idempotent render)
@channel in 50K: Rate-limit fan-out, stagger over 1-2s to avoid thundering herd
Cross-region lag: EU user may see msg 80ms after US user — acceptable for chat

Data Model (Vitess/MySQL)

Sharded by workspace_id — all data for a workspace lives on one shard

-- Messages table (per workspace shard)
CREATE TABLE messages (
  workspace_id   BIGINT NOT NULL,    -- shard key
  channel_id     BIGINT NOT NULL,
  channel_seq    BIGINT NOT NULL,    -- monotonic per channel
  user_id        BIGINT NOT NULL,
  content        TEXT,
  ts             TIMESTAMP(6),
  PRIMARY KEY (workspace_id, channel_id, channel_seq)
);

-- Channel membership
CREATE TABLE channel_members (
  workspace_id   BIGINT NOT NULL,
  channel_id     BIGINT NOT NULL,
  user_id        BIGINT NOT NULL,
  joined_at      TIMESTAMP,
  PRIMARY KEY (workspace_id, channel_id, user_id)
);

-- Redis (not MySQL) for hot path:
-- channel:{id}:online    → SET of user_ids currently online
-- user:{id}:gateway      → STRING "gateway-host-7"
-- channel:{id}:members   → SET of all user_ids (cached from MySQL)

What Makes This Hard

Five specific challenges that make Slack's delivery non-trivial

#ChallengeSolution
150K-member fan-out — can't write 50K times per messageOnly fan-out to online members (~10% of total). Offline catch up on reconnect.
2Ordering — users must see messages in same orderKafka partition per channel + monotonic channel_seq. Client detects gaps.
3Exactly-once delivery — WS connections drop constantlyClient tracks last_seen_seq, requests backfill on reconnect. Dedup by seq.
4Presence at scale — knowing who's online across 10M connectionsGateway reports connect/disconnect to Redis. Presence aggregated per channel.
5Thundering herd — @channel in 50K-member channelRate-limit fan-out, stagger delivery over 1-2 seconds for huge channels.
